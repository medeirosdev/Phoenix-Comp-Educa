Módulo 01

Fundamentos C/C++ e Arduino

Linguagem de programação, tipos, funções, condicionais e como o Arduino funciona.

O que é uma linguagem de programação?

Computadores (e microcontroladores dentro de robôs) trabalham com 0s e 1s — a linguagem binária. Uma linguagem de programação é um tradutor: você escreve código em algo próximo ao inglês/matemática e o compilador converte para os 0s e 1s que o hardware entende.

Quanto mais próxima da linguagem humana, mais "alto nível" é a linguagem. C e C++ são linguagens de nível médio — mais próximas da máquina que Python, mas muito mais legíveis que Assembly.

C vs C++

A linguagem C foi criada nos anos 70 e é a base de grande parte do software embarcado no mundo. C++ é uma extensão do C que adiciona recursos como Orientação a Objetos — pense nela como um C com ferramentas extras.

No Arduino (e no firmware da Bia), usamos C++ com o framework Arduino, que adiciona funções específicas para hardware como digitalRead(), analogWrite() e delay().

Compilação: em C/C++, o código precisa ser compilado antes de rodar. O compilador lê todo o código, checa erros e gera um binário. Se tiver um erro de sintaxe, você descobre na compilação — antes de subir para o robô.

Variáveis e Tipos

Variáveis armazenam valores que o programa usa. Em C/C++, você deve declarar o tipo da variável antes de usá-la:

int    contador      = 0;       // número inteiro: -2.147B a +2.147B
float  velocidade    = 1.5;     // número decimal (ponto flutuante)
bool   calibrado     = false;   // verdadeiro ou falso
char   letra         = 'A';     // um único caractere
uint8_t pwm_valor   = 200;     // inteiro 8 bits sem sinal: 0 a 255

No firmware do robô você vai encontrar muito float (ganhos do PID, velocidades) e uint8_t/int (leituras de sensores, contadores).

Funções

Funções são blocos de código reutilizáveis. Em vez de repetir o mesmo trecho 10 vezes, você cria uma função e chama ela quantas vezes quiser:

// Declara a função — tipo_retorno nome(parâmetros)
int somar(int a, int b) {
    return a + b;
}

// Usando a função
int resultado = somar(3, 4);  // resultado = 7

Funções que não retornam nada usam o tipo void. É muito comum no Arduino: void setup() e void loop() são funções void obrigatórias.

Condicionais (if / else / switch)

if (posicao > 4000) {
    virar_direita();
} else if (posicao < 4000) {
    virar_esquerda();
} else {
    ir_reto();
}

// switch é útil para múltiplos casos discretos
switch (modo_atual) {
    case LINHA:   run_line_follower(); break;
    case VIRTUAL: run_pure_pursuit(); break;
    default:      parar(); break;
}

Loops (while / for)

// while — executa enquanto a condição for verdadeira
while (bateria_ok) {
    loop_principal();
}

// for — ideal quando se sabe quantas vezes repetir
for (int i = 0; i < NUM_SENSORES; i++) {
    leituras[i] = analogRead(pinos[i]);
}

Arduino: setup() e loop()

O framework Arduino exige duas funções especiais:

  • setup() — executa uma vez ao ligar o robô. Aqui você configura pinos, inicia comunicação Serial, calibra sensores.
  • loop() — executa repetidamente enquanto o robô estiver ligado. É o coração do programa.
void setup() {
    Serial.begin(115200);
    pinMode(LED, OUTPUT);
}

void loop() {
    float pos = ler_posicao_linha();
    float correcao = calcular_pid(pos);
    aplicar_motores(correcao);
}

Ferramentas do Desenvolvedor

Git para controle de versão e PlatformIO para desenvolver firmware embarcado.

Git — Controle de Versão

Git é um sistema que rastreia as mudanças no seu código ao longo do tempo. Pense nele como um "histórico com checkpoint" do seu projeto: você pode voltar a qualquer versão anterior, trabalhar em funcionalidades paralelas sem bagunçar o código principal e colaborar com a equipe.

Conceitos fundamentais

  • Repositório: a pasta do projeto, com todo o histórico de mudanças.
  • Commit: um snapshot salvo localmente. Como um "save game".
  • Branch: uma linha de desenvolvimento paralela. Você cria uma branch para desenvolver uma feature sem afetar o código principal (main).
  • Push: envia seus commits locais para o servidor (GitHub).
  • Pull: baixa as últimas mudanças do servidor.

Fluxo básico de trabalho

# 1. Baixar o repositório para sua máquina
git clone https://github.com/phoenix-unicamp/seguidor-linha.git

# 2. Criar uma branch para sua feature
git checkout -b minha-feature

# 3. Editar código... depois salvar as mudanças
git add arquivo.cpp
git commit -m "feat: adiciona calibração automática"

# 4. Enviar para o GitHub
git push origin minha-feature
Regra de ouro: nunca commite direto na branch main. Crie sempre uma branch, desenvolva, teste, depois faça um Pull Request para o time revisar.

PlatformIO — IDE para Embarcados

O PlatformIO é uma ferramenta que roda dentro do VSCode e facilita muito o desenvolvimento para microcontroladores. Ele gerencia as bibliotecas, compila o código e faz o upload para o hardware automaticamente.

Por que usar PlatformIO e não Arduino IDE?

  • Gerenciamento de dependências via platformio.ini
  • Suporte nativo ao ESP32-S3 (chip do nosso robô)
  • Integração com Git e VSCode
  • Build system mais rápido e configurável

Arquivo platformio.ini (exemplo da Bia)

[env:esp32s3]
platform  = espressif32
board     = esp32-s3-devkitc-1
framework = arduino

lib_deps =
    bluetooth
    ESP32Encoder

Comandos essenciais

  • Build (✓): compila o projeto, verifica erros
  • Upload (→): compila e grava no microcontrolador via USB
  • Monitor (🔌): abre o terminal Serial para ver logs do robô

Sensores de Linha

Como o robô "vê" a pista — sensores ópticos, leitura analógica e cálculo de posição.

O Sensor Óptico-Reflexivo

Os sensores de linha do robô são pares óptico-reflexivos: cada sensor contém um LED emissor de luz infravermelha e um fototransistor receptor. O LED emite luz; o fototransistor mede quanta luz voltou refletida da superfície abaixo.

Por que funciona em preto e branco?

Superfícies brancas refletem quase toda a luz → fototransistor recebe muito → tensão alta → analogRead() retorna valor alto (~3000–4095).

Superfícies pretas absorvem quase toda a luz → fototransistor recebe pouco → tensão baixa → analogRead() retorna valor baixo (~0–500).

Por isso a pista precisa de alto contraste. Sem ele, o sensor não consegue distinguir linha de superfície.

Array de Sensores

Um único sensor diz apenas "tem linha aqui ou não". Para saber onde na pista o robô está, usamos vários sensores em linha horizontal (S0, S1, … S8). A Bia tem 9 sensores frontais.

Lendo Posição com Média Ponderada

Cada sensor tem uma posição fixa no array. Para converter 9 leituras em uma única posição contínua, usamos média ponderada:

// Sensores: S0=0, S1=1000, S2=2000, ... S8=8000
// 4000 = centro da pista

float soma_pesos = 0;
float soma_leituras = 0;

for (int i = 0; i < NUM_SENSORES; i++) {
    soma_pesos    += leitura[i] * (i * 1000.0f);
    soma_leituras += leitura[i];
}

float posicao = soma_pesos / soma_leituras;  // 0 a 8000

Resultado: posição 0 = robô na extrema esquerda, 4000 = centro, 8000 = extrema direita. O erro é posicao - 4000.

Calibração

Os valores brutos variam por iluminação e variações de hardware. A calibração resolve isso: o robô passa os sensores sobre branco puro e preto puro, salva os extremos, e depois normaliza tudo para 0–100%.

// Após calibração: normaliza para 0-1000
int normalizar(int raw, int minVal, int maxVal) {
    return map(raw, minVal, maxVal, 0, 1000);
}

Motores e Controle Básico

PWM, controle diferencial e a limitação do controle por if/else.

PWM — Controlando Velocidade

PWM (Pulse Width Modulation) é a técnica usada para controlar a velocidade dos motores. Em vez de alterar a tensão, o sinal alterna entre ligado e desligado muito rapidamente. O motor "vê" uma tensão média proporcional ao tempo ligado:

  • 50% duty cycle → ~50% da velocidade máxima
  • 100% duty cycle → velocidade máxima
  • 0% duty cycle → motor parado
// analogWrite envia PWM para o motor (0-255 no Arduino)
analogWrite(MOTOR_ESQUERDO, 180);   // ~70% de velocidade
analogWrite(MOTOR_DIREITO,   180);   // igual → vai reto

Controle Diferencial

O robô não tem volante — ele vira diferenciando a velocidade dos dois motores. Motor esquerdo mais rápido que o direito → vira à direita. Motor direito mais rápido → vira à esquerda.

// Para virar à direita: motor esquerdo mais rápido
analogWrite(MOTOR_ESQUERDO, 220);
analogWrite(MOTOR_DIREITO,  120);

Seguindo com If/Else

A primeira tentativa de seguir a linha é usar if/else baseado nos sensores:

int pos = ler_posicao();

if (pos > 4500) {
    // Robô desviou para a direita → virar esquerda
    analogWrite(MOTOR_ESQUERDO, 120);
    analogWrite(MOTOR_DIREITO,  220);
} else if (pos < 3500) {
    // Robô desviou para a esquerda → virar direita
    analogWrite(MOTOR_ESQUERDO, 220);
    analogWrite(MOTOR_DIREITO,  120);
} else {
    analogWrite(MOTOR_ESQUERDO, 180);
    analogWrite(MOTOR_DIREITO,  180);
}
Problema: o controle if/else é "tudo ou nada". O robô alterna abruptamente entre "vira muito" e "vai reto", causando zigue-zague. Para velocidades maiores, isso é catastrófico. A solução é o controle contínuo — PID.

Normalização da Posição

Antes de enviar para o controlador, normalizamos a posição para uma escala centrada em zero (erro = posição − centro):

// Erro centralizado: -4000 a +4000, sendo 0 = centrado
float erro = posicao_normalizada - 4000.0f;

// Positivo = desvio para a direita, negativo = para a esquerda

Controle PID

O algoritmo central do robô seguidor — como o PD transforma erro em correção suave.

Por que PID?

O controle if/else gera zigue-zague porque responde de forma binária. O PID calcula uma correção proporcional ao erro — quanto maior o desvio, maior a correção. O resultado é um movimento suave e preciso.

As Três Parcelas

PID significa Proporcional, Integral, Derivativo:

  • P — Proporcional: reage ao erro atual. Erro grande → correção grande.
  • I — Integral: reage ao erro acumulado. Corrige desvios persistentes.
  • D — Derivativo: reage à velocidade de mudança do erro. Amortece oscilações.

Por que usamos PD (sem o I)?

Em seguidores de linha, a pista curva constantemente. O erro muda o tempo todo, então um acumulador de erro (I) ficaria sempre errado. Ficamos com PD: proporcional + derivativo.

O Algoritmo em Código

float kP = 0.8f;     // ganho proporcional
float kD = 5.0f;     // ganho derivativo
float vel_base = 180; // velocidade quando erro = 0

float erro_anterior = 0;

void loop() {
    float erro = ler_posicao() - 4000.0f;

    float P = kP * erro;
    float D = kD * (erro - erro_anterior);
    float correcao = P + D;

    analogWrite(MOTOR_ESQUERDO, vel_base + correcao);
    analogWrite(MOTOR_DIREITO,  vel_base - correcao);

    erro_anterior = erro;
}
Interpretação: correcao positiva → robô desviou para a direita → motor esquerdo acelera, direito freia → robô retorna ao centro. Se negativa, o inverso.

Sintonizando kP e kD

Não há uma fórmula mágica — a sintonização é empírica. Estratégia:

  1. Começa com kP pequeno (ex: 0.3), kD = 0. O robô segue a linha mas oscila.
  2. Aumenta kD até as oscilações diminuírem.
  3. Aumenta kP para o robô reagir mais rápido nas curvas.
  4. Se oscilar muito → aumenta kD. Se reagir devagar → aumenta kP.

Sinais de kP/kD errados

  • kP muito alto: robô oscila em zigue-zague constante
  • kD muito alto: robô treme, reage a qualquer variação mínima
  • kP muito baixo: robô não corrige rápido o suficiente nas curvas
