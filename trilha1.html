Fundamentos C/C++ e Arduino

Linguagem de programação, tipos, funções, condicionais e como o Arduino funciona.

O que é um Programa de Computador?

Computadores, microcontroladores, smartphones — toda máquina digital opera com o mesmo princípio básico: ela só entende dois estados elétricos, que representamos como 0 (sem tensão) e 1 (com tensão). A partir desses dois símbolos, toda a computação do planeta é construída.

Um processador moderno executa bilhões de instruções por segundo, e cada instrução é uma sequência de 0s e 1s. Escrever programas diretamente nessa linguagem binária seria humanamente impossível. Por isso existem as linguagens de programação — elas são tradutores entre a nossa linguagem (palavras, símbolos matemáticos, lógica) e a linguagem da máquina.

Quanto mais a linguagem se parece com o inglês ou com matemática, mais alto nível ela é. Python é uma linguagem de alto nível: você escreve print("olá") e ela cuida de tudo. Assembly é quase a linguagem da máquina, praticamente ilegível para humanos. C e C++ ficam no meio: são legíveis o suficiente para programar com produtividade, mas próximos o suficiente do hardware para controlar cada detalhe — o que é essencial em robótica.

Por que C/C++ e não Python? Microcontroladores como o ESP32-S3 da Bia têm ~512KB de RAM e precisam responder a eventos em microssegundos. C/C++ gera código compacto e previsível. Python é ~10–100x mais lento e usa muito mais memória. Para robótica embarcada, C/C++ é a escolha padrão na indústria.

A Linguagem C e a Linguagem C++

A linguagem C foi criada em 1972 por Dennis Ritchie nos Bell Labs (mesma empresa onde nasceu o Unix). É uma das linguagens mais influentes já criadas — o kernel do Linux, o Windows, grande parte do firmware de equipamentos médicos, automotivos e aeroespaciais são escritos em C.

A linguagem C++ foi criada em 1979 por Bjarne Stroustrup como uma extensão do C. Ela adiciona:

  • Classes e objetos (Programação Orientada a Objetos)
  • Templates (código genérico para múltiplos tipos)
  • Sobrecarga de operadores (definir o que + faz para seus tipos)
  • Biblioteca padrão rica (std::vector, std::string, etc.)
  • Gerenciamento de memória mais seguro (referências, RAII)

Todo código C válido também é C++ válido (com raras exceções). Pense em C++ como C com muitas ferramentas extras — você pode usá-las quando quiser e ignorá-las quando não precisar.

No firmware da Bia usamos C++ com o framework Arduino. O Arduino adiciona uma camada de funções específicas para hardware, tornando mais fácil ler sensores, controlar pinos e comunicar via Serial ou BLE.

Como o Código Vira um Robô: A Compilação

C/C++ são linguagens compiladas. Isso significa que antes de qualquer coisa acontecer no hardware, o código-fonte passa por um processo de transformação chamado compilação. Entender esse processo vai te ajudar a interpretar erros e depurar problemas.

Passo 1 — Preprocessamento

O preprocessador (cpp) trata todas as linhas que começam com #. Ele substitui #include pelo conteúdo dos arquivos de cabeçalho, expande #define e processa diretivas condicionais (#ifdef, #endif). O resultado é um arquivo C++ puro, sem diretivas.

Passo 2 — Compilação

O compilador (g++ ou clang++) lê o C++ pré-processado e o converte em código Assembly — instruções específicas do processador, mas ainda legíveis por humanos. Aqui acontece também a verificação de tipos, a otimização do código e a detecção de erros de sintaxe e semântica.

Passo 3 — Montagem (Assembly)

O montador (assembler) converte o Assembly em código objeto — bytes binários organizados em seções (código, dados, etc.). Ainda não é executável: faltam as referências a funções de outras bibliotecas.

Passo 4 — Ligação (Linking)

O ligador (linker) junta todos os arquivos objeto e as bibliotecas necessárias em um único binário executável. Para o ESP32, esse binário é um arquivo .bin ou .elf que será gravado na memória flash do microcontrolador.

Passo 5 — Upload

O PlatformIO conecta ao ESP32 via USB e grava o binário na memória flash. O microcontrolador reinicia e começa a executar o código imediatamente.

Erros de compilação vs erros em tempo de execução: erros de sintaxe (falta um ponto e vírgula, chave não fechada) são detectados na compilação — antes de qualquer coisa rodar. Erros lógicos (divisão por zero, acesso fora de um array) ocorrem em tempo de execução. Compilação é um filtro poderoso, mas não pega tudo.

Variáveis e Tipos de Dados

Uma variável é um espaço reservado na memória RAM com um nome que você escolhe. Quando você declara int contador = 0;, o compilador reserva 4 bytes na RAM, associa o nome contador a esse endereço e armazena o valor 0 lá.

Em C/C++, toda variável tem um tipo definido em tempo de compilação. O tipo determina: quantos bytes são usados, que valores cabem, que operações são permitidas. Você não pode guardar um número decimal em um int sem perda de informação.

Tipos Inteiros

int           x = 42;        // 4 bytes — de -2.147.483.648 a +2.147.483.647
unsigned int  u = 3000;      // 4 bytes — de 0 a 4.294.967.295 (sem sinal)
short         s = -100;     // 2 bytes — de -32.768 a +32.767
long          l = 100000;   // 4 bytes no ESP32 (mesmo que int)
long long     ll = 9999999999; // 8 bytes — números enormes

Tipos Inteiros de Tamanho Fixo (recomendados em embedded)

No ESP32 e outros microcontroladores, o tamanho de int pode variar entre plataformas. Para ter certeza absoluta do tamanho, use os tipos fixos definidos em <stdint.h>:

uint8_t   sensor_raw = 200;    // 1 byte sem sinal: 0 a 255
int8_t    erro_pequeno = -5;   // 1 byte com sinal: -128 a 127
uint16_t  adc_leitura = 3800;  // 2 bytes sem sinal: 0 a 65.535
int16_t   posicao = -1500;    // 2 bytes com sinal: -32.768 a 32.767
uint32_t  timestamp = 12000000; // 4 bytes: 0 a ~4.29 bilhões

No firmware da Bia você vai encontrar muito uint8_t para valores de PWM (0–255), uint16_t para leituras ADC dos sensores (0–4095), e uint32_t para timestamps em microssegundos.

Tipos Ponto Flutuante

float   kp = 1.5f;       // 4 bytes — ~7 dígitos de precisão
double  pi = 3.14159265; // 8 bytes — ~15 dígitos de precisão

float é o padrão em embedded — double usa o dobro de memória e no ESP32 não há unidade de ponto flutuante de dupla precisão (seria lento). Use sempre float. O sufixo f em 1.5f indica explicitamente que é float, não double.

Tipo Booleano

bool  calibrado   = false;   // falso
bool  motor_ligado = true;   // verdadeiro
// Em C, qualquer valor != 0 é considerado true
bool  resultado = (5 > 3);   // true, pois 5 > 3 é verdadeiro

Tipo Caractere

char  letra = 'A';           // 1 byte — armazena um caractere ASCII
char  newline = '\n';        // caracteres especiais com barra invertida
// Strings são arrays de char terminados em '\0'
char  nome[] = "Phoenix";    // 8 bytes: P,h,o,e,n,i,x,\0

Constantes

Constantes são valores que não mudam durante a execução. Há duas formas de defini-las:

// 1. Usando const — tem tipo definido, checado pelo compilador (preferido)
const int   NUM_SENSORES = 9;
const float PI_VALOR     = 3.14159f;

// 2. Usando #define — substituição textual pelo preprocessador
#define MAX_PWM 255
#define MOTOR_ESQUERDO 16   // pino GPIO do motor

Prefira const — o compilador verifica o tipo e pode otimizar melhor. Use #define para números de pinos e macros do framework Arduino.

Nomeando Variáveis

Um bom nome de variável é autoexplicativo — quem lê o código entende sem precisar de comentário. Na Phoenix usamos snake_case (palavras separadas por underscore):

// Ruim: o que é 'x'? o que é 'v'?
float x = 0;
int   v = 180;

// Bom: nomes longos são bem-vindos!
float posicao_normalizada = 0;
int   velocidade_base_pwm = 180;
bool  giroscopio_calibrado = false;

Operadores

Operadores são os símbolos que realizam operações sobre variáveis e valores. C/C++ tem categorias bem definidas:

Operadores Aritméticos

int a = 10, b = 3;
int soma      = a + b;   // 13
int diferenca = a - b;   // 7
int produto   = a * b;   // 30
int quociente = a / b;   // 3  ← ATENÇÃO: divisão inteira!
int resto     = a % b;   // 1  (módulo: resto da divisão)
Armadilha — Divisão Inteira: quando dois int são divididos, o resultado é sempre inteiro (o decimal é descartado). 10 / 3 = 3, não 3.333. Para obter o decimal, ao menos um dos operandos precisa ser float:

float resultado = (float)10 / 3; → 3.333...
float resultado = 10.0f / 3; → 3.333...

Esse é um dos erros mais frequentes em C para quem vem de outras linguagens.

Type Casting — Conversão de Tipo

Você pode forçar a conversão de um tipo para outro com cast:

int   adc_leitura = 3800;
float percentual  = (float)adc_leitura / 4095.0f * 100.0f;
// Sem o (float), seria divisão inteira: 3800/4095 = 0!

int pwm_arredondado = (int)velocidade;  // corta o decimal: 180.9 → 180

Operadores de Atribuição Compostos

int contador = 0;
contador += 5;   // equivale a: contador = contador + 5  → 5
contador -= 2;   // equivale a: contador = contador - 2  → 3
contador *= 4;   // equivale a: contador = contador * 4  → 12
contador /= 3;   // equivale a: contador = contador / 3  → 4
contador %= 3;   // equivale a: contador = contador % 3  → 1

Incremento e Decremento

int i = 5;
i++;   // pós-incremento: usa i=5, depois vira 6
++i;   // pré-incremento: vira 7, depois usa i=7
i--;   // pós-decremento: usa i=7, depois vira 6

// Na prática, em loops simples a diferença não importa:
for (int j = 0; j < 9; j++) { /* j++ e ++j têm o mesmo efeito aqui */ }

Operadores de Comparação

int x = 10, y = 5;
x == y   // false  — igual a?
x != y   // true   — diferente de?
x > y    // true   — maior que?
x < y    // false  — menor que?
x >= y   // true   — maior ou igual a?
x <= y   // false  — menor ou igual a?
Erro clássico — = vs ==: = é atribuição, == é comparação. Escrever if (x = 5) em vez de if (x == 5) não dá erro de compilação em C, mas o comportamento é completamente diferente: x = 5 atribui 5 a x e retorna 5 (verdadeiro). O robô vai fazer coisas inesperadas. Sempre revise seus ifs.

Operadores Lógicos

bool a = true, b = false;

a && b   // false — AND: verdadeiro só se AMBOS forem verdadeiros
a || b   // true  — OR:  verdadeiro se PELO MENOS UM for verdadeiro
!a       // false — NOT: inverte o valor lógico

// Exemplo prático:
if (bateria_ok && sensores_calibrados) {
    iniciar_corrida();
}
if (!linha_detectada || posicao > 8000) {
    parar_emergencia();
}

Funções

Funções são blocos de código com um nome, que você pode chamar de qualquer lugar do programa. Elas são o principal mecanismo de organização em C/C++. Sem funções, você teria que repetir o mesmo código em todos os lugares que precisasse dele — e se precisasse corrigir um bug, teria que corrigir em todos os lugares.

Anatomia de uma função

// tipo_retorno  nome_da_funcao (tipo param1, tipo param2, ...)
float  calcular_erro(float posicao_atual, float posicao_alvo) {
    float erro = posicao_atual - posicao_alvo;
    return erro;  // retorna o resultado para quem chamou
}
  • Tipo de retorno (float): o tipo do valor que a função devolve. Use void se não retornar nada.
  • Nome (calcular_erro): o identificador. Use verbos que descrevem o que ela faz.
  • Parâmetros (float posicao_atual, float posicao_alvo): os dados de entrada. São copiados — modificar o parâmetro dentro da função não afeta a variável original (passagem por valor).
  • Corpo: o código entre chaves { }.
  • return: devolve o resultado e encerra a função imediatamente.

Funções void (sem retorno)

void ligar_led(int pino) {
    digitalWrite(pino, HIGH);
    // não precisa de return (pode ter um "return;" vazio para sair cedo)
}

void parar_motores() {   // zero parâmetros também é válido
    analogWrite(MOTOR_ESQ, 0);
    analogWrite(MOTOR_DIR, 0);
}

Protótipos (declaração antecipada)

Em C/C++, você precisa declarar uma função antes de usá-la. Se calcular_pid() chama ler_sensores(), mas ler_sensores está definida depois no arquivo, o compilador vai reclamar. A solução é o protótipo: uma declaração sem corpo no topo do arquivo.

// Protótipos no topo — só a assinatura, sem o corpo
float ler_posicao_linha();
void  aplicar_motores(float correcao);

void loop() {
    float pos = ler_posicao_linha();   // compila pois o protótipo já foi visto
    aplicar_motores(pos - 4000);
}

float ler_posicao_linha() { /* ... implementação ... */ }

Múltiplos parâmetros e múltiplos return

// Função com vários parâmetros
int limitar(int valor, int minimo, int maximo) {
    if (valor < minimo) return minimo;
    if (valor > maximo) return maximo;
    return valor;
}

// Chamada:
int pwm = limitar(correcao_calculada, -255, 255);

Escopo de Variáveis

O escopo define onde uma variável existe e pode ser acessada. É delimitado pelos { }.

Variáveis locais

void minha_funcao() {
    int local = 10;   // existe só dentro desta função
    // ...
}   // 'local' é destruída aqui

// Erro: 'local' não existe aqui fora
// int x = local;  → erro de compilação

Variáveis globais

// Declaradas fora de qualquer função — visíveis em todo o arquivo
float g_velocidade_base = 180.0f;  // prefixo g_ indica global
bool  g_corrida_ativa   = false;

void iniciar_corrida() {
    g_corrida_ativa = true;          // acessa a global
}

void loop() {
    if (g_corrida_ativa) { /* ... */ }
}
Quando usar global vs local? Use local sempre que possível — a variável só existe onde é necessária e não pode ser modificada acidentalmente por outras funções. Use global para estado compartilhado entre múltiplas funções que precisam ler e escrever o mesmo valor — como o estado da corrida, os parâmetros do PID, a posição do robô.

Variável static local

// static local: existe para sempre, mas só é visível dentro da função
void calcular_derivada(float erro_atual) {
    static float erro_anterior = 0.0f; // inicializada só uma vez
    float derivada = erro_atual - erro_anterior;
    erro_anterior = erro_atual;         // salva para a próxima chamada
    return;
}

Static local é muito usado no firmware da Bia para manter estado entre chamadas sem usar variável global.

Estruturas Condicionais

Condicionais permitem que o programa tome decisões. Sem elas, o código executaria sempre as mesmas instruções na mesma ordem, independente de qualquer entrada.

if simples

float tensao_bateria = ler_bateria();

if (tensao_bateria < 3.5f) {
    Serial.println("ALERTA: bateria fraca!");
    parar_motores();
}

if / else

float posicao = ler_posicao_linha();

if (posicao > 4000) {
    // robô está à direita da linha → vira esquerda
    ajustar_motores(-50);
} else {
    // robô está à esquerda da linha → vira direita
    ajustar_motores(50);
}

if / else if / else (cadeia)

float erro = posicao - 4000.0f;

if      (erro > 2000)  { ajustar_motores(-200); }  // muito à direita
else if (erro > 500)   { ajustar_motores(-80); }   // levemente à direita
else if (erro > -500)  { ajustar_motores(0); }    // centrado — vai reto
else if (erro > -2000) { ajustar_motores(80); }    // levemente à esquerda
else                   { ajustar_motores(200); }    // muito à esquerda

switch / case

Quando você precisa comparar uma variável inteira com vários valores fixos, switch é mais legível que uma cadeia de if/else if:

switch (modo_corrida) {
    case MODO_LINE_FOLLOWER:
        run_pid_controller();
        break;   // OBRIGATÓRIO — sem break, executa o próximo case também!

    case MODO_VIRTUAL_LINE:
        run_pure_pursuit();
        break;

    case MODO_PARADO:
        parar_motores();
        break;

    default:   // executado se nenhum case combinar
        Serial.println("Modo desconhecido!");
        break;
}

Laços de Repetição

Loops permitem executar um bloco de código múltiplas vezes. São essenciais para percorrer arrays de sensores, aguardar condições, ou executar cálculos iterativos.

while — enquanto a condição for verdadeira

// Aguarda calibração ser concluída
while (!calibracao_concluida()) {
    processar_calibracao();
    delay(10);
}

// Loop infinito (comum em sistemas embarcados)
while (true) {
    executar_sistema();
}

for — ideal quando se sabe o número de iterações

O for tem três partes separadas por ponto e vírgula:

//      inicialização ; condição    ; incremento
for (int i = 0   ; i < NUM_SENSORES ; i++) {
    leituras[i] = analogRead(pinos_sensores[i]);
}

// Equivalente em while — o for é apenas mais compacto:
int i = 0;
while (i < NUM_SENSORES) {
    leituras[i] = analogRead(pinos_sensores[i]);
    i++;
}

do-while — executa ao menos uma vez

// Lê sensor e repete se leitura for inválida
int leitura;
do {
    leitura = analogRead(SENSOR_FRENTE);
} while (leitura < 0 || leitura > 4095); // repete se inválido

break e continue

// break: interrompe o loop imediatamente
for (int i = 0; i < 100; i++) {
    if (linha_perdida()) {
        break;   // sai do loop agora
    }
    processar(i);
}

// continue: pula para a próxima iteração
for (int i = 0; i < NUM_SENSORES; i++) {
    if (leituras[i] < 100) {
        continue;   // ignora sensores com leitura muito baixa
    }
    processar_sensor(i);
}

Arrays (Vetores)

Um array é uma coleção de variáveis do mesmo tipo armazenadas em sequência na memória, acessadas por um índice numérico. Em C/C++, o primeiro elemento tem índice 0.

Declaração e acesso

// Declara array de 9 inteiros (não inicializado — lixo de memória!)
int leituras[9];

// Declara e inicializa
int pinos[9] = {34, 35, 36, 37, 38, 39, 40, 41, 42};

// Acesso: índice 0 = primeiro elemento
leituras[0] = analogRead(pinos[0]);  // sensor mais à esquerda
leituras[8] = analogRead(pinos[8]);  // sensor mais à direita

// Percorrer com for é a forma natural:
for (int i = 0; i < 9; i++) {
    leituras[i] = analogRead(pinos[i]);
}
Perigo — acesso fora dos limites: acessar leituras[9] em um array de tamanho 9 é um comportamento indefinido — o programa lê ou escreve em memória que pertence a outra variável. C/C++ não checa limites automaticamente. Sempre garanta que seu índice está entre 0 e tamanho-1.

O Framework Arduino

O Arduino não é só hardware — é também uma API de programação (um conjunto de funções) que simplifica o controle de microcontroladores. Na equipe, usamos o hardware do ESP32-S3, mas a linguagem e as funções são as mesmas do Arduino.

Estrutura obrigatória: setup() e loop()

void setup() {
    // Executa UMA VEZ ao ligar ou resetar
    // Aqui: configurar pinos, iniciar serial, calibrar sensores
    Serial.begin(115200);                 // abre porta serial a 115200 baud
    pinMode(LED_BUILTIN, OUTPUT);         // configura pino como saída
    Serial.println("Sistema iniciado!");
}

void loop() {
    // Executa CONTINUAMENTE enquanto ligado
    // Aqui: ler sensores, calcular PID, mover motores
    float pos = ler_posicao_linha();
    float correcao = calcular_pid(pos);
    mover_motores(correcao);
}

Pinos digitais

// Configura pino como entrada ou saída
pinMode(13, OUTPUT);       // pino 13 como saída
pinMode(2,  INPUT);        // pino 2 como entrada
pinMode(4,  INPUT_PULLUP); // entrada com resistor pull-up interno

// Escrita digital — coloca o pino em HIGH (3.3V) ou LOW (0V)
digitalWrite(13, HIGH);  // acende LED
digitalWrite(13, LOW);   // apaga LED

// Leitura digital — retorna HIGH (1) ou LOW (0)
int estado = digitalRead(2);
if (estado == HIGH) { Serial.println("Botão pressionado"); }

Leitura analógica

// analogRead retorna 0 a 4095 (ADC de 12 bits no ESP32)
int valor_bruto = analogRead(34);  // lê pino 34
// 0 = 0V, 4095 = 3.3V (tensão máxima)

// Converter para porcentagem:
float pct = (float)valor_bruto / 4095.0f * 100.0f;

Escrita analógica (PWM)

// analogWrite gera sinal PWM no pino — 0 a 255
analogWrite(16, 0);    // motor parado
analogWrite(16, 128);  // ~50% de velocidade
analogWrite(16, 255);  // velocidade máxima
// No ESP32 com PlatformIO pode-se usar ledcWrite para mais controle

Tempo e temporização

// delay(ms) — bloqueia o programa por N milissegundos
delay(1000);  // para TUDO por 1 segundo — evite em código de produção!

// millis() — milissegundos desde o boot (não bloqueia)
uint32_t agora = millis();

// Padrão não-bloqueante: faz algo a cada 100ms
static uint32_t ultimo_envio = 0;
if (millis() - ultimo_envio >= 100) {
    enviar_telemetria();
    ultimo_envio = millis();
}

// micros() — microssegundos (maior precisão)
uint32_t t_inicio = micros();
processar_algo();
uint32_t duracao = micros() - t_inicio;  // quanto tempo levou em µs
Por que evitar delay()? Enquanto o delay() está rodando, o microcontrolador não faz nada — não lê sensores, não responde a BLE, não processa nenhum outro código. No loop principal do robô (~2ms por ciclo), um único delay(100) destruiria completamente a performance. Use sempre millis() para temporizações não-bloqueantes.

Funções utilitárias do Arduino

// map — mapeia um valor de um intervalo para outro
int angulo = map(2048, 0, 4095, 0, 180);  // retorna 90
// map(valor, fromLow, fromHigh, toLow, toHigh)

// constrain — limita um valor a um intervalo
int pwm = constrain(correcao, -255, 255);  // não deixa sair de ±255

// abs — valor absoluto
float magnitude = abs(-3.14f);  // 3.14

// max/min
int maior = max(10, 20);  // 20
int menor = min(10, 20);  // 10

Serial — depuração via cabo USB

void setup() {
    Serial.begin(115200);   // inicia comunicação serial a 115200 baud
}

void loop() {
    float pos = ler_posicao();

    Serial.print("pos: ");       // imprime sem nova linha
    Serial.println(pos);         // imprime COM nova linha
    Serial.printf("pos=%.2f erro=%.2f\n", pos, pos - 4000);  // formatado
}

No PlatformIO, abra o monitor serial com o ícone de plugue elétrico (ou pio device monitor no terminal). Configure o baud rate para 115200.

Juntando tudo: estrutura típica de um arquivo Arduino

// ── Includes ─────────────────────────────
#include <Arduino.h>

// ── Defines e constantes ─────────────────
#define MOTOR_ESQ 16
#define MOTOR_DIR 17
const int NUM_SENSORES = 9;

// ── Variáveis globais ─────────────────────
float    g_kp = 1.2f;
float    g_kd = 6.0f;
uint16_t g_leituras[NUM_SENSORES];

// ── Protótipos ────────────────────────────
void  ler_sensores();
float calcular_posicao();

Ferramentas do Desenvolvedor

Git para controle de versão e PlatformIO para desenvolver firmware embarcado.

Git — Controle de Versão

Git é um sistema de controle de versão distribuído criado por Linus Torvalds em 2005 (o mesmo que criou o Linux). Ele rastreia todas as mudanças no seu código ao longo do tempo, com data, autor e mensagem explicativa. Pense nele como um "histórico com checkpoint" completo do projeto: você pode voltar a qualquer momento anterior, ver exatamente o que mudou e por quê, e trabalhar em múltiplas funcionalidades em paralelo sem interferir no código estável.

Conceitos fundamentais

  • Repositório (repo): a pasta do projeto com todo o histórico. Existe uma cópia local (no seu computador) e uma remota (no GitHub).
  • Commit: um snapshot salvo localmente — como um "save game". Cada commit tem um ID único (hash), uma mensagem e um autor.
  • Branch: uma linha de desenvolvimento paralela. A branch main é o código estável. Você cria branches para desenvolver features novas sem quebrar o que já funciona.
  • Merge: une o conteúdo de uma branch na outra. Geralmente feito via Pull Request no GitHub.
  • Push: envia seus commits locais para o GitHub.
  • Pull: baixa e integra as últimas mudanças do GitHub para o seu computador.
  • Clone: baixa um repositório remoto inteiro (com todo o histórico) pela primeira vez.

Comandos do dia a dia

# Ver o que exatamente mudou nos arquivos

# Adicionar arquivos ao próximo commit (staging)

git add src/              # adiciona uma pasta inteira

# Salvar um checkpoint local com mensagem descritiva

# Ver histórico de commits

Trabalhando com branches

PlatformIO — IDE para Embarcados

O PlatformIO é uma extensão do VSCode (e também uma ferramenta de linha de comando) que foi criada especificamente para desenvolvimento de sistemas embarcados. Ele resolve três problemas que a Arduino IDE não resolve bem: gerenciamento de dependências, suporte a plataformas diversas (ESP32, STM32, AVR, etc.) e integração com Git e ferramentas profissionais.

Por que PlatformIO e não Arduino IDE?

  • Gerenciamento de bibliotecas: as dependências ficam declaradas no platformio.ini e são baixadas automaticamente. Em qualquer máquina, pio run baixa tudo e compila.
  • Suporte ao ESP32-S3: suporte nativo e atualizado ao chip da Bia.
  • Integração com VSCode: autocomplete, navegação de código, debugging integrado.
  • Build system robusto: compila apenas o que mudou, muito mais rápido.
  • Múltiplos ambientes: você pode ter um env:debug com flags de debug e um env:release otimizado no mesmo projeto.

Estrutura de um projeto PlatformIO

├── include/

Sensores de Linha

Como o robô "vê" a pista — sensores ópticos, leitura analógica e cálculo de posição.

O Sensor Óptico-Reflexivo

Os sensores de linha são pares óptico-reflexivos — dispositivos que combinam um emissor e um receptor de luz em um único encapsulamento. O emissor é um LED de infravermelho (IR, ~950nm, invisível ao olho humano). O receptor é um fototransistor que varia sua condutância elétrica de acordo com a intensidade da luz IR que recebe.

O princípio de funcionamento é simples: o LED emite luz para baixo. Parte dessa luz é refletida pela superfície sob o sensor e volta ao fototransistor. Quanto mais luz retorna, maior a tensão na saída do sensor — e essa tensão é lida pelo ADC do microcontrolador via analogRead().

O contraste de reflexão: preto vs branco

A chave do funcionamento está nas propriedades ópticas das cores:

  • Superfícies brancas são altamente reflexivas → devolvem ~80-90% da luz IR → fototransistor recebe muito → tensão alta → analogRead() retorna ~3000–4095
  • Superfícies pretas são absorvedoras → devolvem ~5-10% da luz IR → fototransistor recebe pouco → tensão baixa → analogRead() retorna ~50–400

A diferença entre os dois extremos (chamada de "swing" do sensor) tipicamente fica entre 2000 e 3500 unidades de ADC. É essa diferença que permite detectar a linha.

Dependência da iluminação ambiente: luz fluorescente e luz solar direta emitem IR e afetam as leituras. Em competições, o ambiente de iluminação varia muito. Por isso a calibração antes de cada corrida é obrigatória — ela ajusta os valores de referência para as condições do momento.

Por que múltiplos sensores?

Um único sensor diz apenas "tem linha aqui?" — sim ou não. Com um único sensor, você só sabe se o robô está ou não sobre a linha, sem saber o quanto ele desviou nem para qual lado. Isso é informação insuficiente para um controlador.

Com 9 sensores em linha horizontal, você tem um "campo de visão" que cobre toda a largura da linha mais uma margem de cada lado. Cada sensor reporta a intensidade da reflexão naquele ponto, e a combinação das 9 leituras permite calcular a posição da linha com resolução sub-sensor.

Lendo os Sensores em Código

const int NUM_SENSORES = 9;
const int PINOS[NUM_SENSORES] = {34,35,36,37,38,39,40,41,42};
uint16_t g_leituras[NUM_SENSORES];

void ler_todos_sensores() {
    for (int i = 0; i < NUM_SENSORES; i++) {
        g_leituras[i] = (uint16_t)analogRead(PINOS[i]);
    }
}

// S0=80  S1=90  S2=100 S3=400 S4=3800 S5=420 S6=110 S7=95  S8=85

Calculando a Posição com Média Ponderada

A sacada da média ponderada é tratar cada sensor como um "peso" que atrai o resultado para sua posição. Atribuímos posições fixas: S0=0, S1=1000, S2=2000, ..., S8=8000. O centro da pista é 4000.

A fórmula: posição = (soma de leitura_i × posição_i) / (soma de leitura_i)

Quando a linha está sob S4 (posição 4000), a leitura de S4 é alta e "puxa" o resultado para 4000. Quando a linha está entre S3 e S4, as leituras intermediárias dos dois sensores geram um valor entre 3000 e 4000 — você obtém precisão contínua, não apenas 9 posições discretas.

float calcular_posicao() {
    float soma_ponderada = 0.0f;
    float soma_leituras  = 0.0f;

    for (int i = 0; i < NUM_SENSORES; i++) {
        soma_ponderada += (float)g_leituras[i] * (i * 1000.0f);
        soma_leituras  += (float)g_leituras[i];
    }

    if (soma_leituras < 200.0f) {
        return -1.0f;  // linha não detectada — todos sensores no preto
    }

    return soma_ponderada / soma_leituras;  // 0 a 8000
}

// Exemplos numéricos:
// S4=3800, demais≈0  → pos = (3800×4000)/3800 = 4000 (centro exato)
// S3=2300, S4=1500   → pos = (2300×3000 + 1500×4000)/(2300+1500) = 3447 (entre S3 e S4)

Calibração dos Sensores

As leituras brutas dos sensores variam com a iluminação ambiente, distância ao chão, envelhecimento dos LEDs e diferenças de fabricação. A calibração resolve isso capturando os valores extremos e normalizando tudo para uma escala padrão.

Algoritmo de calibração

uint16_t g_cal_min[NUM_SENSORES];  // mínimo (leitura sobre PRETO)
uint16_t g_cal_max[NUM_SENSORES];  // máximo (leitura sobre BRANCO)

void iniciar_calibracao() {
    for (int i = 0; i < NUM_SENSORES; i++) {
        g_cal_min[i] = 4095;   // começa com o máximo possível
        g_cal_max[i] = 0;      // começa com o mínimo possível
    }
}

void amostrar_calibracao() {
    for (int i = 0; i < NUM_SENSORES; i++) {
        uint16_t leitura = (uint16_t)analogRead(PINOS[i]);
        if (leitura < g_cal_min[i]) g_cal_min[i] = leitura;
        if (leitura > g_cal_max[i]) g_cal_max[i] = leitura;
    }
}

// Normaliza leitura bruta para 0-1000 usando os extremos calibrados
int normalizar(int i, uint16_t raw) {
    int span = g_cal_max[i] - g_cal_min[i];
    if (span < 100) return 0;  // sensor sem contraste — ignora
    int norm = (int)((float)(raw - g_cal_min[i]) / span * 1000);
    return constrain(norm, 0, 1000);
}

Detecção de linha perdida

bool linha_detectada() {
    for (int i = 0; i < NUM_SENSORES; i++) {
        // Se qualquer sensor lê acima de 500 → há linha em algum lugar
        if (normalizar(i, g_leituras[i]) > 500) return true;
    }
    return false;  // todos sensores sobre preto → linha perdida
}

Motores e Controle Básico

PWM, controle diferencial e a limitação do controle por if/else.

PWM — Controlando a Velocidade dos Motores

Motores DC funcionam com tensão contínua: tensão máxima = velocidade máxima. Mas como variar a velocidade sem um conversor de tensão analógico (que seria caro e ineficiente)? A resposta é o PWM — Pulse Width Modulation (Modulação por Largura de Pulso).

Em vez de mudar a tensão, o PWM liga e desliga o sinal em alta frequência (~20kHz no ESP32). O motor possui inércia mecânica suficiente para não "perceber" as oscilações — ele integra o sinal e reage à tensão média. O parâmetro que controlamos é o duty cycle: a porcentagem do tempo em que o sinal está em HIGH.

  • 0% duty cycle: sinal sempre LOW → tensão média = 0V → motor parado
  • 50% duty cycle: metade do tempo HIGH → tensão média = metade do máximo → ~50% da velocidade
  • 100% duty cycle: sinal sempre HIGH → tensão média = máxima → velocidade máxima
// analogWrite: 0=parado, 255=máximo (mapeado para 0-100% duty cycle)
analogWrite(MOTOR_ESQUERDO, 0);    // parado
analogWrite(MOTOR_ESQUERDO, 128);  // ~50% de velocidade
analogWrite(MOTOR_ESQUERDO, 255);  // velocidade máxima

// Ambos iguais → robô vai reto
analogWrite(MOTOR_ESQUERDO, 180);
analogWrite(MOTOR_DIREITO,  180);

O Driver de Motor — H-Bridge

O microcontrolador não pode alimentar o motor diretamente — a corrente necessária (centenas de mA) destruiria os pinos GPIO. Por isso usamos um driver de motor, tipicamente um H-bridge (ponte H).

O H-bridge é um circuito que pode conectar o motor na direção desejada (frente ou ré) com corrente vinda de uma fonte separada (a bateria do robô). O ESP32 envia apenas o sinal de controle; toda a potência vem da bateria. No firmware, você controla direção com dois pinos digitais e velocidade com o PWM.

// Esquema típico: INA/INB controlam direção, PWM controla velocidade
digitalWrite(MOTOR_ESQ_INA, HIGH);
digitalWrite(MOTOR_ESQ_INB, LOW);   // direção: frente
analogWrite (MOTOR_ESQ_PWM, 180);  // velocidade

Controle Diferencial — como o robô vira

O robô seguidor não tem volante nem servo. Ele gira diferenciando a velocidade dos dois motores — assim como um tanque de guerra. Este princípio chama-se acionamento diferencial.

  • Ambos motores mesma velocidade: robô vai reto
  • Motor esquerdo mais rápido: robô gira para a direita (o lado mais lento "arrasta")
  • Motor direito mais rápido: robô gira para a esquerda
  • Velocidades opostas: robô gira no próprio eixo (pirouette)
// Traduzindo: correction > 0 significa "desviei para a direita"
// → preciso girar à esquerda → motor esquerdo desacelera, direito acelera
void aplicar_motores(int vel_base, float correcao) {
    int esq = constrain(vel_base - (int)correcao, 0, 255);
    int dir = constrain(vel_base + (int)correcao, 0, 255);
    analogWrite(MOTOR_ESQ_PWM, esq);
    analogWrite(MOTOR_DIR_PWM, dir);
}

Controle por if/else — a primeira tentativa

A estratégia mais intuitiva para seguir a linha é usar condicionais baseados na posição dos sensores. Você define zonas e determina o que fazer em cada uma:

float posicao = calcular_posicao();  // 0 a 8000, 4000=centro

if (posicao > 5000) {
    // Muito à direita da linha → vira agressivamente à esquerda
    analogWrite(MOTOR_ESQ_PWM, 100);
    analogWrite(MOTOR_DIR_PWM, 220);
} else if (posicao > 4200) {
    // Levemente à direita → correção suave
    analogWrite(MOTOR_ESQ_PWM, 150);
    analogWrite(MOTOR_DIR_PWM, 200);
} else if (posicao >= 3800) {
    // Centrado → vai reto na velocidade base
    analogWrite(MOTOR_ESQ_PWM, 180);
    analogWrite(MOTOR_DIR_PWM, 180);
} else if (posicao >= 3000) {
    analogWrite(MOTOR_ESQ_PWM, 200);
    analogWrite(MOTOR_DIR_PWM, 150);
} else {
    analogWrite(MOTOR_ESQ_PWM, 220);
    analogWrite(MOTOR_DIR_PWM, 100);
}
O problema fundamental do if/else: o controle é discreto — ele salta entre valores fixos de correção. Quando o robô está no limite entre duas zonas (posição oscilando entre 4199 e 4201), ele alterna abruptamente entre duas correções diferentes a cada ciclo. O resultado é zigue-zague. Em velocidades baixas (≤ 1m/s) isso funciona. Em velocidades de competição (3-5m/s), o robô oscila e perde a linha. O próximo módulo apresenta a solução: o controlador PD.

Erro de Posição — Preparando para o PID

O controlador PID trabalha com uma grandeza chamada erro: a diferença entre onde o robô está e onde deveria estar (o centro da linha, posição 4000).

// Erro positivo = robô desviou para a direita da linha
// Erro negativo = robô desviou para a esquerda da linha
// Erro zero = robô perfeitamente centrado
float erro = calcular_posicao() - 4000.0f;

// Faixa típica: -4000 a +4000
// -4000: linha na extrema esquerda (sensor S0)
// +4000: linha na extrema direita  (sensor S8)

Com esse erro calculado, o controlador PD gera uma correção contínua e proporcional que vai direto para os motores — sem zonas, sem saltos bruscos.

Controle PID

O algoritmo central do robô seguidor — como o PD transforma erro em correção suave.

Da Resposta Binária ao Controle Contínuo

O problema do if/else é que ele é discreto: existem apenas N respostas possíveis, e o robô salta abruptamente entre elas. O que precisamos é de um controle contínuo: uma resposta que varia suavemente em função do erro, sem saltos.

Imagine dirigir um carro: quando você está levemente saindo da faixa, faz uma pequena correção no volante. Quando está desviando muito, faz uma correção grande. Você não tem apenas dois estados ("virar máximo" ou "não virar") — a correção é proporcional ao quanto saiu da faixa. Isso é exatamente o que o controlador PID faz.

O Controlador PID — Três Perspectivas do Erro

PID significa Proporcional, Integral, Derivativo. Cada letra representa uma perspectiva diferente de análise do mesmo sinal de erro:

P — Proporcional: "o erro agora é grande ou pequeno?"

A parcela proporcional gera uma resposta diretamente proporcional ao erro atual. Se o erro dobra, a correção P dobra. É a parcela mais intuitiva e é a principal responsável pela correção.

float P = kP * erro;
// erro = 2000 (robô muito à direita) → P = 1.2 * 2000 = 2400
// erro = 200  (levemente à direita)  → P = 1.2 * 200  = 240
// erro = 0    (centrado)             → P = 1.2 * 0    = 0

Com apenas a parcela P: o robô segue a linha, mas oscila em torno dela. Ele ultrapassa o centro (overshoot), corrige, ultrapassa de volta — fica em zigue-zague. Para reduzir as oscilações, precisamos da parcela D.

I — Integral: "o erro persiste por muito tempo?"

A parcela integral acumula o erro ao longo do tempo. Se o robô ficar consistentemente desviado por muitos ciclos, o acumulador cresce e força uma correção crescente até eliminar o desvio persistente. É útil para sistemas com atrito ou forças externas constantes.

static float acumulador = 0.0f;
acumulador += erro;           // cresce a cada ciclo
float I = kI * acumulador;

Por que descartamos o I nos seguidores de linha?

Em um seguidor de linha, a pista curva constantemente. O "set point" (posição ideal) é sempre a linha — mas o que "correto" significa muda a cada centímetro. O acumulador integral acabaria integrando erros de quando o robô estava em curvas (que são temporários e necessários) e geraria uma força espúria quando o robô retorna à reta. O efeito é exatamente o oposto do desejado: instabilidade. Ficamos com PD.

D — Derivativo: "o erro está crescendo rápido ou devagar?"

A parcela derivativa reage à velocidade de mudança do erro — a derivada em relação ao tempo. Se o erro está aumentando rapidamente (o robô está saindo da linha em alta velocidade), o termo D gera uma correção oposta imediatamente, antes que o erro fique enorme. É um "freio preditivo".

// Derivada discreta: aproximação por diferenças finitas
float D = kD * (erro_atual - erro_anterior);
// Se o erro cresceu muito desde o último ciclo → D é grande → freia a correção
// Se o erro está estável → D ≈ 0 → não interfere
// Se o erro está diminuindo (robô voltando ao centro) → D é negativo → ajuda

Intuitivamente: a parcela P diz "corrija proporcionalmente ao erro", a parcela D diz "mas se você já está voltando para o centro, não corrija tanto". O D amortece as oscilações que o P gera.

O Algoritmo PD Completo

// ── Parâmetros (ajustáveis via BLE no robô real) ──────────
float g_kP       = 1.2f;   // ganho proporcional
float g_kD       = 8.0f;   // ganho derivativo
float g_vel_base = 180;    // PWM quando erro = 0 (0-255)

// ── Estado persistente entre ciclos ──────────────────────
static float s_erro_anterior = 0.0f;

// ── Função chamada a cada ciclo (~2ms) ────────────────────
void executar_pid() {
    // 1. Lê e calcula o erro atual
    float posicao = calcular_posicao();
    float erro    = posicao - 4000.0f;

    // 2. Calcula as parcelas
    float P = g_kP * erro;
    float D = g_kD * (erro - s_erro_anterior);

    // 3. Correção total
    float correcao = P + D;

    // 4. Aplica nos motores com saturação
    int esq = constrain((int)(g_vel_base - correcao), 0, 255);
    int dir = constrain((int)(g_vel_base + correcao), 0, 255);
    analogWrite(MOTOR_ESQ_PWM, esq);
    analogWrite(MOTOR_DIR_PWM, dir);

    // 5. Salva erro para o próximo ciclo (cálculo do D)
    s_erro_anterior = erro;
}
Por que subtrai do esquerdo e soma no direito? Erro positivo = robô desviou para a direita (linha está à esquerda do centro). Para corrigir, precisamos girar à esquerda: motor direito acelera, motor esquerdo desacelera. Logo: motor_dir = base + correcao, motor_esq = base - correcao. Se a correção for negativa (robô desviou para a esquerda), os sinais invertem automaticamente.

Entendendo o Papel de Cada Parâmetro

Velocidade base (vel_base)

É a velocidade de equilíbrio — o que o robô mantém quando erro = 0 (perfeitamente centrado). Quanto maior, mais rápido o robô anda nas retas, mas as curvas ficam mais difíceis de gerenciar. Comece com ~50-60% do máximo (120-160 de 255).

kP — ganho proporcional

Controla a agressividade da resposta ao erro. Um kP maior faz o robô reagir mais rápido a desvios, mas se for alto demais, ultrapassa o centro e gera oscilação. Um kP baixo demais e o robô não corrige rápido o suficiente nas curvas e sai da pista.

// kP = 0.3: correção suave, pode perder curvas fechadas
// kP = 1.2: correção moderada — bom ponto de partida
// kP = 3.0: correção agressiva — provável oscilação sem kD alto

kD — ganho derivativo

Amortece as oscilações causadas pelo P. Um kD adequado faz o robô "deslizar" suavemente de volta ao centro. Um kD muito alto causa "tremor" — o robô reage exageradamente a pequenas variações do sensor, incluindo ruído eletrônico.

// kD = 0:   sem amortecimento — oscila com kP razoável
// kD = 8:   amortecimento moderado — bom ponto de partida
// kD = 40:  amortecimento excessivo — robô treme na reta

Metodologia de Sintonização (Tuning)

Não existe fórmula mágica. A sintonização é iterativa e empírica — você testa, observa o comportamento, ajusta. Siga esta sequência:

  1. Zera tudo: kP = 0.3, kD = 0, velocidade baixa (~100 PWM). O robô deve seguir a linha mas oscilar.
  2. Aumenta kP gradualmente até o robô seguir bem nas retas mas ainda oscilar nas curvas.
  3. Aumenta kD até as oscilações diminuírem sem causar tremor.
  4. Aumenta velocidade base e repita: com velocidade maior, o robô sente a linha com menos tempo por ciclo, então geralmente precisa de kP e kD maiores.
  5. Teste curvas fechadas especificamente — elas são o caso extremo.

Diagnóstico por comportamento

// SINTOMA → CAUSA → AÇÃO
// ──────────────────────────────────────────────────────────
// Oscila em zigue-zague na reta    → kP alto demais  → reduz kP ou aumenta kD
// Não segue curvas fechadas        → kP baixo demais → aumenta kP
// Treme/vibra na reta              → kD alto demais  → reduz kD
// Lento para corrigir desvios      → kP baixo demais → aumenta kP
// Sai voando em curvas (muito rápido) → vel_base alta → reduz vel_base

Visualizando o PD com Telemetria

O app Phoenix tem uma tela de telemetria que plota o erro em tempo real via BLE a 10Hz. Use-a para diagnosticar o comportamento: o sinal deve oscilar suavemente em torno de zero nas retas e ter picos moderados nas curvas. Oscilações de alta frequência indicam kD excessivo; desvio crescente sem retornar indica kP insuficiente.

// Exemplo de código de telemetria no firmware:
static uint32_t ultimo_telem = 0;
if (millis() - ultimo_telem >= 100) {  // 10Hz
    char buf[64];
    snprintf(buf, sizeof(buf), "T,%.1f,%.1f\r", erro, correcao);
    send_bluetooth_message(buf);
    ultimo_telem = millis();
}
